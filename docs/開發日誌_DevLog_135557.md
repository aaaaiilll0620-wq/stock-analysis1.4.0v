> 目標不是預測大盤漲跌，而是建立一套**可驗證、無未來函數、具鑑別力**的個股評級與策略框架。

## 1. 🛠️ 系統核心模組架構 (System Architecture)

| 層級 / 模組 | 職責 |
|---|---|
| **Data Layer** (`data_provider`) | 串接 FinMind API，進行 **PIT 嚴格時間軸對齊**：價格/PER/月營收只取 `date ≤ as_of`；季報依約 45 天公告時差過濾，回測日尚未公告的季報一律不用。杜絕未來函數 (Look-ahead bias)。 |
| **Cache Layer** (`data_cache` + `build_cache`) | 每檔每資料集一個 Parquet（唯讀 / 只追加 / 保 PIT）作原始快取；回測與個股分析改讀本機 = **0 API**，每天只補增量。查詢層以 **DuckDB** 直接查同一批 Parquet（`duck_query()` + `tbl()`），做跨股橫向 SQL 選股。 |
| **ScoringManager** (五維度評分大腦) | 將個股訊號量化為五大維度分數（見下）。 |
| **ValuationEngine** | 三重視角估值：PEG 成長調整 / 河流圖歷史百分位 / 絕對門檻；並交叉驗證「成長溢價 vs 昂貴泡泡」。 |
| **FundamentalEngine** | 硬門檻安全檢查 + 獲利品質一致性（增收是否增利）+ 現金流健康（OCF/FCF）。 |
| **InvestmentAdvisor** (決策導師) | 讀取五維度分數與各維度 Gate（閘門），套用動態權重與雙軌評級邏輯，輸出個股最終評級與可執行建議。 |
| **ScoreStore** (`score_store`) | 把每檔算好的**五維綜合分**落地成 `Scores` 快取（每檔每日每模式一列），供 DuckDB **跨股排名選股**。評分邏輯完全複用 ScoringManager / InvestmentAdvisor，本層只負責「落地 + 查詢」。 |
| **Backtester** (回測引擎) | PIT 滾動回測，支援固定持有 (Horizon) 與**路徑相依動態出場** (dynamic_stop)；產出分桶報酬、多空價差、權益曲線、市場中性檢驗、參數網格搜尋、因子歸因。 |
| **Web App** (`app.py`, Streamlit) | 「個股分析 / 多檔排行」即時分析 + 「綜合分選股」讀快取跨股排名。公開站支援**每訪客用自己的 FinMind token**（per-session 隔離）。 |

### 五維度評分 (ScoringManager)

| 維度 | 內容 |
|---|---|
| 🏢 **基本面 (Fundamental)** | 營收 / EPS / 三率、負債比、流動比、現金流健康、總資產週轉率（v4.4）。 |
| 💰 **估值面 (Valuation)** | PER / PBR 歷史百分位、PEG、殖利率。 |
| 📈 **技術面 (Technical)** | 均線多頭排列、週線趨勢、RSI 位階、MACD、布林、乖離率。 |
| 🐋 **籌碼面 (Whale)** | 千張大戶持股週變化、法人連續買超力道、近期淨流向、投信吸籌比。 |
| 🚀 **動能面 (Momentum)** | 中期價格動能、漲跌幅 / 量能爆發 / 乖離健康度、月營收年增與加速度、相對大盤強弱 RS（v4.4）。 |

## 3. 🧱 v1.0.0 — 防守型價值選股與僵化評級 (起步期)

**設計思維**：個股分析極度注重安全性。`InvestmentAdvisor` 設立了極其嚴格的過濾閘門，
必須**同時**通過才給【強烈推薦】：

- `g_估值OK`：估值必須「偏低」或「合理」。
- `g_RSI不過熱`：`rsi < rsi_extreme`。
- `g_不追高`：`ma20_bias ≤ bias_chase`。
- 外加品質、分數門檻、籌碼不弱。

**評級桶（硬編碼 3 級）**：`["強烈推薦", "觀望追蹤", "謹慎避開"]`
**回測設定**：固定持有 20 天 (Horizon Exit)。

> **埋下的種子**：這三道閘門要求「便宜 + 不過熱 + 不追高」**同時成立**——
> 而任何一檔漲上去的主流飆股，本質上就是「貴 + RSI 高 + 乖離大」。
> 換言之，**恐高與逆勢被寫進了評級邏輯本身**。

## 4. ⚠️ v1 實測診斷 — 反向指標的重大轉折

### 📉 回測整體數據
- 策略總報酬 **+81.7%**（年化 +22.7%），期勝率 63%，但嚴重落後大盤 0050 的 **+119.9%**（超額報酬 **−38.2%**）。
- 【強烈推薦】分桶平均報酬為負 **−0.58%**，多空價差 **−2.83%** — 系統呈現嚴重的**「反向指標」偏誤**。

### 🔍 個股分析盲點診斷

**① 主流飆股的「恐高症」錯失**

| 個股 | 回測期平均報酬 | 被系統判定 | 問題 |
|---|---|---|---|
| **1519 華城** | **+8.76%** | 35 期中 **26 次【觀望追蹤】** | 漲高→估值爆表、RSI 過熱，飆速段完全沒吃到 |
| **2330 台積電** | +3.59% | 【強勢/強推】次數近乎 0 | 因「不夠便宜」長期卡【觀望】 |
| **3017 奇鋐 / 3324 雙鴻 / 3661 世芯-KY** | +2~6% | 多數【觀望】 | AI／權值核心飆股被估值門檻擋下 |

**② 弱勢反彈股的「反向打臉」**

| 個股 | 回測期平均報酬 | 被系統判定 | 問題 |
|---|---|---|---|
| **2408 南亞科** | **+4.67%** | **31 次【謹慎避開】** | 基本面差被瘋狂避開，卻因底部籌碼流入 + 技術低檔反彈大漲 |

### 🩺 診斷結論
1. 個股分析過於死守「防守閘門」→ 牛市中投資組合全是**死魚股**。
2. 固定持有 20 天 → 強勢股利潤集體回吐。
3. 避開桶藏著贏家 → **多空價差為負**，評級與未來報酬呈反向關係。

## 5. 🚀 v2.0.0 — 四級標籤與中性偏積極動能特赦 (結構重構)

為一次解決「回測落後」與「個股分析失真」，進行大刀闊斧重構。

### A. 個股分析模組 (InvestmentAdvisor) 邏輯翻修

**1. 新增【強勢買進】判定軌道（雙軌制）**
- 保留【強烈推薦】給傳統「低估值、安全基期、基本面好」的長線價值股。
- 新增【強勢買進】(第 4 級) 專收「順勢動能飆股」：當動能強、主力法人狂吸、技術面多頭排列時，
  採用**【中性偏積極的條件式特赦】**，放寬估值百分位與 RSI/乖離限制，
  正式將台積電、華城等主流股納入最高評級。
- **泡沫與衰竭保護**：本益比歷史高檔且成長跟不上（昂貴泡泡）、或價漲量縮 + 主力同步出貨（衰竭）→ 不特赦。

**2. 修正【謹慎避開】的逆勢盲點（逆勢救援）**
- 若個股因基本面差被歸類為避開，但觸發「技術面翻多/突破壓力」且「籌碼面顯著流入」→
  強制提升為【觀望追蹤】，把南亞科型反彈股移出避開桶，消除反向指標。

**3. 動態權重 (Dynamic Weighting)**
- 當個股處於強勢多頭（多頭排列 + 正乖離 + RSI 偏多）時，
  自動把**估值權重砍 60%** 轉給動能與籌碼 → 分數由順勢動能主導，不再被「便宜與否」拖累主流股。

### B. 回測與統計結構重構
1. **硬編碼評級擴充為 4 級**：`{"謹慎避開":0, "觀望追蹤":1, "強烈推薦":2, "強勢買進":3}`。
   全面同步修改 `summarize()`、`equity_curve()`、`track_rating_changes()`、`_bucket_spread()` 的統計/繪圖/排序邏輯。
2. **多空價差修正**：公式改為 **(強勢買進平均報酬 − 謹慎避開平均報酬)**。
3. **全面匹配動態出場**：`run()` 預設 `exit_mode="dynamic_stop"` —— 收盤實質跌破量價成本區 (POC/Support)
   或觸發 σ 移動停利 (Chandelier) 立刻提早出場，**拒絕死抱 20 天**（合成回測平均持有降至約 11 天）。
4. **五維度權重重配（中性偏積極）**：技術/動能/籌碼合計拉高至約 70%，估值退居安全邊際確認。

### 📊 v2.0.0 實測
| 指標 | v1 | v2.0.0 |
|---|---|---|
| 多空價差 | −2.83% | **+0.44%**（轉正） |
| 強勢買進桶報酬 | 不存在 | **+2.21%（最高桶，67 樣本）** |
| 主流飆股歸類 | 全卡觀望 | 台積電 14 次強勢、聯發科 12、廣達 9、世芯 5 |
| 權益曲線 (集中買進級) | +81.7% / −13.6% | **+109.1% / −9.8% 回檔** |

## 6. 🔬 v2.1 — 依 432 筆關卡診斷的資料驅動校準

啟用**每次都印的兩軌關卡未過率診斷**後，用 432 筆真實樣本定位瓶頸，逐一對症下藥。

### 發現的四個問題與修正

| # | 問題（實測瓶頸） | 修正 |
|---|---|---|
| **1（最關鍵）** | **致命排序 bug**：「估值偏高 + 綜合分<50」的軟避開被排在強勢買進判定**之前**。主流飆股綜合分常被估值拖到 50 以下 → 還沒輪到強勢買進就被錯殺成避開。 | 改判定順序為：**硬避開（基本面/現金流/破線）→ 強勢買進 → 估值型軟避開**。 |
| **2** | 動能夠強 69.9% 未過：`momentum_score` 是單日快照、太噪。 | 動能改**三選一**：價量動能分 **或** 技術結構分 (多頭排列·MACD·週線) **或** 營收動能。 |
| **3** | 籌碼點火 79.2% 未過：`whale_score` 計分天生偏低（需連續買超才拉得高）。 | 加**替代點火路徑**：有實質流入且（whale 達標 **或 土洋同步連買 或 法人成交占比≥40%**）。 |
| **4** | 多頭排列 61.1% 未過：`ma5≥ma20` 在健康回檔會短暫下彎，錯殺回測到月線的主流股。 | 多頭排列改以「**站上月線 + 站穩週線 + 中期未死叉**」認定，不綁噪動的 5 日線。 |

外加門檻下修：`whale_hot 45→42`、`mom_hot 50→46`、`min_score`（balanced 62→58）。

## 7. ✅ v3.0 — 驗證與生產配置定案

### 7.1 決定性證據：市場中性樣本外驗證

用 `--validate` 把「絕對報酬」與「市場中性（剃除大盤 beta）」兩條路攤開對比：

| 檢驗 | train | test | 判定 |
|---|---|---|---|
| 絕對報酬（只買強推） | −0.58% | +1.21% | ❌ 未同時為正 |
| 絕對報酬（強推+觀望） | +0.24% | −1.57% | ❌ 未同時為正 |
| **市場中性（同日 前⅓ − 後⅓）** | **+0.96%** | **+3.23%** | ✅ **通過（且樣本外放大）** |

> **關鍵洞察**：系統**會「排序」（哪些強、哪些弱），但不會「擇時」（大盤何時漲跌）**。
> 當初的「反向指標」本質是**把「不會擇時」誤當成「選股選反了」**——
> 剃掉大盤 beta 後，高分股確實贏低分股（test **+3.23%**），且 out-of-sample 站得住。
> **重構在數據上被驗證成功。**

市場中性對沖曲線與大盤相關性 **+0.07**（近乎零）→ 確認是純粹的排序 α，不靠大盤。

### 7.2 配置實驗：集中 vs 分散（反直覺結論）

既然排序力已驗證，實測各種「怎麼用它」的配置（同區間、同基準）：

| 配置 | 總報酬 | 最大回檔 | Calmar | 判讀 |
|---|---|---|---|---|
| **買 強勢+強推（分散，等權）** | **+109.1%** | **−9.8%** | **2.66** | ✅ **最佳** |
| 純做多前⅓ (market-neutral 系列) | +113.2% | −24.8% | 1.19 | 報酬相近、回檔大 |
| 排序 top5（分數加權） | +96.8% | −23.6% | 1.11 | 集中反而變差 |
| 排序 top5（等權） | +85.0% | −19.3% | 1.22 | |
| 排序 top3（分數加權） | +61.5% | −23.6% | 0.76 | 越集中越差 |
| 多空對沖（前⅓−後⅓） | +60.0% | −24.3% | 0.72 | 市場中性、corr 0.07 |
| 大盤 0050 | +119.9% | −16.2% | 1.91 | benchmark |

> **反直覺結論：集中化（排序 top-N）全面變差**——報酬更低、回檔更大，越集中越糟。
>
> **系統真正的超能力不是「排序」，是「沒好貨時自動空手」**：排序法每期強迫買前 N 名，
> 大盤修正時仍滿倉吃回檔；而分散買買進級的版本，當沒幾檔通過「多頭排列＋籌碼＋基本面」時
> 自然少買/空手 → 修正時自動降曝險 → 回檔壓到大盤的一半（−9.8% vs −16.2%）。

```bash
python tests\run_backtest.py           # 買 強勢買進+強烈推薦、等權、dynamic_stop 出場
```

- **總報酬 +109.1%、最大回檔 −9.8%、Calmar 2.66（風險調整後勝 0050 的 1.91）。**

## 9. 🌐 v4.0 — 快取層、跨股綜合分選股與網頁化上線

v1→v3 建立並驗證了「個股評級 + 回測」，v3.1 將其工具化。**v4.0 把系統再推進兩個新能力**：
(1) 從「一次看一檔」→「一次對**整個名單**依五維綜合分**跨股排名**選股」；
(2) 從「本機自用」→「**可公開部署、別人各用自己 FinMind token** 的網站」。

### A. 本機資料快取層（`core/data_cache.py` + `build_cache.py`）
- 每檔每資料集一個 Parquet（唯讀 / 只追加 / 保 PIT）；回測與個股分析改讀本機 = **0 API**，每天只補增量。
- 查詢層：**DuckDB** 直接查同一批 Parquet（`duck_query()` + `tbl()`），可組任意跨股橫向 SQL。
- 讀寫穿透快取代理 `CachingDataLoader`：包住 FinMind DataLoader，查詢自動落地並重用。
- `build_cache.py`：限流並行建庫 / 每日增量；`--screen` 示範用**原始欄位**（PER 等）跨股初篩。

### B. 綜合分物化層（`core/score_store.py`）
- **痛點**：原始欄位（PER / 殖利率 / 法人）DuckDB 查得到，但**五維綜合分**是一整條 pipeline 算出來的，SQL 無法重現。
- **解法**：把每檔算好的 composite **落地**成一張 `Scores` 快取（每檔每日每模式一列），再用 DuckDB 排序篩選。
- 評分邏輯**完全複用** ScoringManager + InvestmentAdvisor（與 `backtest._score_one` 同一套 pipeline），本層只負責「落地 + 查詢」。
- **三模式 = 一張表 + `mode` 欄**（不拆三張表）；每列附 `weights_version`（權重改版可辨識該重算的列）。
- 查詢 API：`screen_by_composite()`（跨股排名 + 橫斷面百分位）、`latest_scores()`、`universe_info()`、`cached_symbols()`。
- 儲存沿用 `data_cache` 的 Parquet 慣例（`Scores/<stock_id>.parquet`，去重鍵 `(as_of, mode)`），`tbl('Scores')` 直接可查。

### C. 日更一步到位（`build_cache.py` 擴充）
- `python build_cache.py --build-scores`：讀本機原始快取算五維綜合分並落地（0 API）。
- `python build_cache.py --screen-composite`：跨股綜合分排名示範。
- 一般 `python build_cache.py`：原始資料增量更新後**自動刷新當天 scores**（0 API）；`--no-scores` 可關。
  scores 步驟獨立 try，失敗不影響已完成的原始快取更新。

### D. 網頁選股分頁（`app.py` 擴充）
- Streamlit 新增「🎯 綜合分選股」分頁：讀 scores 快取對**整個名單**依綜合分排名，可篩門檻 / 評級 / 資料信心、
  顯示**橫斷面百分位**、CSV 下載、重新載入快取。
- 依側欄策略模式查對應 `mode`；快取未建時**友善提示**（不報錯）。
- 原「個股分析 / 多檔排行」維持即時分析，兩者互補（即時看特定股 vs 快取排整個名單）。

### E. 綜合分排名有效性驗證（`validate_scores.py`）
- 補現有回測工具沒有的**綜合分完整分位數階梯**：每期把股票按綜合分切 Q1..Q5，看各桶後續報酬 / 勝率是否**階梯狀遞增**。
- 另給市場中性多空（最高桶 − 最低桶逐期平均）、階梯單調性、綜合 **Rank IC / ICIR / IC>0 佔比**。
- universe **預設對齊 scores 快取名單**（驗證的名單 = 網頁選股的名單）；0 API，結果存 `outputs/validation/`。
- 更深入的 train/test 樣本外、留一貢獻、五維各自 IC 仍走 `tests/run_backtest.py --attribution / --validate / --cycle`。

### F. 公開站「B 模式」多使用者 token 隔離（`app.py`）
- **目標**：公開站讓**每個訪客用自己的 FinMind token**（額度算訪客的、不共用你的）。
- **陷阱**：`DataProvider` 是類別層級單例（`_api` / `_logged_in` 都在 class 上），而 Streamlit 一個程序服務所有訪客 →
  直接對全域登入會讓並發訪客互相蓋掉 token。
- **解法**：每個 token 建一個獨立、已包快取的 loader；分析時在 `threading.Lock` 內把 `DataProvider._api` 暫換成該 loader、
  `_logged_in=True`（使 `_ensure_login` 短路，**絕不 fallback 去讀伺服器 .env 的你的 token**），跑完 `finally` 還原。
- 選股分頁**免 token**（讀快照）；未貼 token 走匿名。已用**並發模擬**驗證：多訪客交錯呼叫各用各的 loader、全域狀態每次正確還原。

### G. 部署上線資產
- `requirements.txt`（雲端依賴）、`.gitignore` 擴充（排除 `dist/` `build/` `outputs/` `finmind_cache/`，**保留** `cloud_cache/` 快照）。
- `DEPLOY_streamlit_cloud.md`：Streamlit Community Cloud 上線步驟手冊（scores 快照打包、`FINMIND_CACHE` secret、**不放你的 token**）。
- `deploy_scores.py` + `update_and_push.bat`：一鍵「更新 scores 快照 → commit → push」，雲端自動重部署（含「無變化就跳過 commit」與「清除下市檔」）。

> **此節點的意義**：v3 證明「排序力有效」；v4.0 把這個排序力**物化成可跨股查詢的表**、接上**網頁選股**、
> 並打通**公開部署與日常更新**。系統從「本機能證明有效」跨到「別人打開瀏覽器就能用」。

## 11. 🧪 v4.2 — 資料驅動的評分核心翻修：whale 重構、權重重配與 Regime 濾網

起點是藍圖第 1 項「**資料層完整度稽核（最高優先）**」：composite 中位僅 ~41、分數門檻多數未過，
當時假設是「法人買賣天數、週線 MA、月營收在 PIT 重建有缺漏」。本節點的方法論是**先量測、再對症、每步用歸因複驗**——
結果稽核推翻了原假設，並把問題一路追到計分設計與市場 regime，最終完成三項核心翻修。

### A. 資料層完整度稽核（`audit_data_completeness.py`，0 API）

新增三段式稽核工具：Part A 逐檔逐資料集查原始快取覆蓋；Part B 對每檔抽月末 as_of 重跑
`build_pit_stockdata`，量測三訊號缺漏率；Part C 讀 Scores 快取做五維分佈稽核。實測（45 檔 × 12 個月末，540 面板列）：

| 檢查項 | 結果 | 判定 |
|---|---|---|
| 原始快取（8 資料集 × 45 檔） | 全部存在，2019-01~2026-07，關鍵欄位無 NaN | ✅ **資料層乾淨** |
| 週線 MA 退化率 | 0.2% | ✅ 非主嫌 |
| 月營收動能 accel/cum 落 None | 0.0% | ✅ 非主嫌 |
| 法人買賣「雙 0 天」 | **36.9%**（連買天數中位 0/0） | ❌ **破口** |
| Scores 五維中位（balanced） | fund 49 / val 57 / tech 48 / mom 64 / **whale 26（29% 掛 0）** | ❌ whale 唯一低於 composite 中位 47.5 |

> **結論**：「composite 系統性偏低」不是資料缺漏，是**計分設計**——whale 的基底
> 「連續買超天數 × 20」是嚴格連續計數，大型權值股法人買賣交錯 → 天數恆為 0 → 基底塌陷；
> 真正穩健的淨流向資訊反而被壓在 ±25 的 adj 裡浪費。chip 資料是滿的，是計法把好資訊丟了。
> 另注意：composite 絕對中位偏低對**跨股排名**幾乎無影響（選股頁用 percent_rank），它咬到的是評級閘門——
> 而「沒幾檔通過就自動空手」正是 v3 驗證過的超能力，故不為美化數字而稀釋門檻。

### B. whale 重構：多天期法人淨參與率當基底，連買天數降級為 bonus

法人買賣依 **1 / 3 / 5 / 10 / 20 天**分開計算「淨參與率」＝該天期（外資＋投信）淨買超 ÷ 同期總成交量
（signed、市值中性、幾乎不為 0），加權（0.10/0.15/0.25/0.25/0.25，斜率 `_RATIO_TO_POINTS=300`）後作 0–100 基底；
原「連續買超天數」cap 3 天、降為 ±12 bonus；**移除 adj 裡與淨參與率共線的重複計分**
（large_holder_activity / foreign_flow / trust_flow），吸籌比 / 法人參與度 / TDCC 保留為確認層（±15 / ±8）。
`backtest`（PIT）與 `data_provider`（live）兩條路同步實作；無多天期資料自動回退舊公式。

> 單元驗證：「連買 0 天但多天期淨流入為正」的大型股，舊版塌到低分 → 新版 **59.8**；
> 持續調節 31 < 交錯淨流入 60 < 強力同步吸籌 100，單調性成立。
> 歸因複驗：whale Rank IC +0.027 → 重構後 +0.027 不變——**量測修乾淨了，但籌碼本質是弱因子**；
> 它的正確待遇是「乾淨版計法 + 降權」，此結論直接餵給下一步的權重重配。

## 12. 🛡️ v4.3 — Bear 段評級閘門與 Regime 偵測穩健化

承接 v4.2 遺留項目「bear 段評級閘門」：v4.2 修好了空頭的「排序」（regime 權重乘數），但「評級」
（強烈推薦／強勢買進）門檻是固定值、不吃 regime，2022 買進桶多空仍 −2.99%。本節點分兩層修：
先讓評級門檻吃 regime，再回頭發現 regime 偵測本身有結構性問題，一併修穩。

### A. Bear 段評級閘門（`core/regime.py::regime_rating_gates`）

比照 `REGIME_MULTIPLIERS` 的權重乘數，新增評級門檻乘數，`InvestmentAdvisor._decide_rating`
呼叫時依 `current_regime` 暫時墊高門檻（`finally` 還原，不污染下一次呼叫）：

| Regime | min_score 加成 | chip_min 乘數 | whale_hot 乘數 |
|---|---|---|---|
| bull / neutral | +0 | ×1.00 | ×1.00 |
| **bear** | **+10** | **×1.50** | **×1.40** |

依 2022 歸因（籌碼在陰跌盤是反指標）只加嚴籌碼相關閘門，`mom_hot`（動能閘門）不動避免錯殺真動能股。
`--cycle` 複驗：**2022 買進桶多空 −2.99% → −2.41%**，排序力不受影響，2021/2023–25 兩段多頭分毫不差。

### B. Regime 偵測滯後與 Whipsaw 診斷（`scripts/diagnose_regime.py`）

逐日跑 `classify_regime` 走過 2021–2025（1209 交易日），發現原始版本 **49 次切換中有 27 次是
whipsaw**（10 個交易日內又切回原狀態），且 2022 空頭確認**滯後 13%**（高點後跌 13% 才判定 bear，
2025 關稅崩盤僅滯後 7% 即確認——兩次滯後程度不一致）。逐一嘗試三個修法，兩個失敗、一個成功：

| 嘗試 | 結果 | 判定 |
|---|---|---|
| 週線多時間框確認 | 對 27 次 whipsaw **零抑制效果**——whipsaw 集中在市場真正橫盤整理區間，週線同樣模糊、不會跟日線意見相左 | ❌ 無效但無害，保留 |
| 波動率正規化深跌破門檻 | 2025 案例反而**更差**：下跌本身推高已實現波動率、門檻被同步放寬，偵測延後至崩盤中段（−17.6% 才觸發，原固定門檻 −7% 即觸發）——內生性缺陷 | ❌ 已預設關閉 |
| **冷卻期/遲滯（K-of-M 密度版）** | whipsaw **27 → 2 次**，切換數 49 → 20 次，2022 排序力 +0.32% → **+0.35%**（略優）、買進桶多空與原始一致 | ✅ **採用** |

冷卻期第一版（純連續天數、對稱）踩到兩個坑：①只把「離開 bear」調慢，沒調「進入 bear」，導致
2022 年底陰跌探底期（原始訊號從未連續撐住 5 天）卡在 neutral 進不去 bear，2022 排序力反而
**轉負至 −0.36%**；②修好進場方向後，敏感度分析又發現「連續 N 天」的判斷方式本身在
`bear_enter_confirm_days=4` 出現真實懸崖（1–3 天結果穩定，4 天直接讓排序力崩潰）——因為連續計數
被單一雜訊日打斷就整個歸零重來。最終改用 **K-of-M 密度版**（近 M 天內至少 K 天同向，不要求不間斷）
取代純連續計數，`K=M` 時完全等同舊版行為（純泛化、非行為改變），且把原本 1–3 安全 / 4 崩潰的窄懸崖
擴大成 **K=1~4（M=5）全部安全**的寬高原：

| 方向 | 參數（預設） | 設計理由 |
|---|---|---|
| 進入 bear | `bear_enter_k=2, bear_enter_m=5`（偏快、寬容） | 防守要快 |
| 離開 bear | `bear_exit_k=10, bear_exit_m=10`（偏慢、等同連續版） | 解除警報要慢 |
| bull ↔ neutral | `confirm_days=5`（連續版，K=M） | 未觀察到懸崖，不需改 |
| 深跌破快速通道 | 豁免冷卻期，一命中立刻確認 bear | 保留急殺即時反應（2025 案例驗證） |

### C. 轉強期對稱實驗：驗證「不用特別處理」是對的

比照 bear 的滯後代價實驗，對稱測試轉強期：2022 秋回升段（低點 2022-10-25 後 3 週 vs 現行
2023-02-10 才確認 bull）。結果排序力僅從 +3.81% 降到 +2.86%（兩種情境皆強 ✅，無翻負），
代價遠低於 bear（bear 提早套用配方會讓排序力由正轉負、翻約 3.7 個百分點）。
印證假設：bull 乘數本來就溫和（動能 1.10×、技術 1.05×），誤判代價低，**轉強期不值得做
不對稱冷卻期**，此結論已用實測數字驗證，非單純猜測。

### D. 樣本代表性調查：已知限制,非本節點引入的新問題

抓一批全新、不重疊的 28 檔（跨產業、8 資料集 × 26 檔 = 208 次 API）獨立測試 `--cycle`：
**2021/2022 排序力雙雙失效**（+2.53%→−0.35%、+0.35%→−0.30%），2023–25 保持但轉弱。三步排查：

1. **排除「檔數少」**：從原始 45 檔隨機抽同尺寸 27 檔，三段全過且排序力更強 → 不是樣本量問題。
2. **排除「特定類型股票是天敵」**：leave-one-out 歸因找拖累最兇的股票，**2021 與 2022 兩段的拖累
   名單幾乎不重疊**（2021 最兇是 3037 欣興，2022 最兇是 1477 聚陽），且移除單一股票都不足以讓
   排序力回到原始 45 檔水準 → 不是少數壞股票拖累,是雜訊疊加。
3. **合併驗證**：45 檔 + 新 26 檔併成 73 檔重跑,三段全部回到 ✅（雖比原始 45 檔單獨跑稀釋）。

**結論**：原始 45 檔測試池刻意做的「跨產業、低相關性」篩選，本身就是排序力穩健度的真實貢獻者，
不只是「涵蓋面廣」的裝飾——換一批沒有刻意篩低相關性的股票，排序力會明顯打折甚至失效。
**這是系統既有的限制（非 v4.3 引入），目前排序力只在原始 45 檔測試池上被充分驗證**；
若未來要把系統用在 45 檔以外的選股，應預期排序力打折，且尚未量化打折幅度——留給未來節點。

**結論**：因子本質沒壞（PEG／橫斷面都是正 IC），是**混合配方**壞——位階成分在多頭是反向指標。
掃描混合比例（70/30、85/15、100/0）：100/0 全期最好但 2022 綜合 −1.34% 明顯變差；
**85/15 是甜蜜點**（估值 IC +0.022→+0.039、單因子 −0.81→−0.26%，並保留位階做「昂貴泡泡」交叉驗證與熊市價值傾斜）。
再降權情境（0.08→0.04/0.00）測過，增益小於重修且犧牲訊號 → **權重維持 0.08、只重修配方**。
