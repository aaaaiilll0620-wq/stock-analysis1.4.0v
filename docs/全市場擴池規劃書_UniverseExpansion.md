# 🌐 全市場擴池規劃書 — TWSE/TPEx 開放資料兩層架構

> **狀態**：規劃階段（可行性已實測驗證，尚未開始實作）
> **日期**：2026-07-13
> **背景**：藍圖第 6 項「擴大 universe」原本卡在 FinMind API 額度（register 等級 ~600 次/小時，
> 全市場 ~1700 檔 × 8 資料集 ≈ 13,600 次一次性建庫、且每日增量仍需 ~6,800 次/天，不可持續）。
> 本規劃書驗證「改用證交所/櫃買中心官方開放資料」是否能繞開這個限制。

---

## 1. 結論先講：可行，但不是原本想像的樣子

**好消息**：TWSE（證交所）與 TPEx（櫃買中心）的官方開放資料平台，六個端點全部實測成功，
免登入、免權杖、無感知限流，一次呼叫就能拿到**全市場**（上市+上櫃合計約 1,700–1,800 檔）的
股價、估值（PE/PBR/殖利率）、三大法人買賣超——這正好對應你系統裡最花 API 的三個「每日純追加」
資料集。

**壞消息（關鍵限制）**：這些端點**只回傳「今天」的快照，完全不支援指定歷史日期**。實測用
`date=20250115` 參數查詢，回傳的仍是最新交易日（2026-07-13）的資料；`openapi.twse.com.tw/v1`
的官方 Swagger 定義裡，這些 `_ALL` 端點也確實沒有 date 參數。

**這代表什麼**：這條路只能拿來做「**從今天開始、每天累積**」的全市場快照，**不能拿來一次補齊
過去幾年的歷史**，也就不能立刻拿它去跑 `--cycle`（2021–2025 三週期）驗證「全市場粗篩規則」
本身的排序力——這件事需要另外用 FinMind 的既有 PIT 能力來做（見 §4 Phase 2 的拆解）。

## 2. 實測驗證結果（2026-07-13 現場逐一打過）

| 交易所 | 端點 | 內容 | 實測結果 |
|---|---|---|---|
| TWSE | `exchangeReport/STOCK_DAY_ALL` | 上市全市場每日 OHLCV | ✅ 一次回傳數百上千檔（含 ETF/個股），CSV 格式 |
| TWSE | `exchangeReport/BWIBBU_ALL` | 上市全市場 PE／殖利率／PBR | ✅ **1,630 檔**，欄位齊全 |
| TWSE | `fund/T86`（三大法人買賣超） | 上市全市場外資/投信/自營商買賣超股數 | ✅ **1,000+ 檔**，拆到外資／外陸資／投信／自營商(自行/避險) 逐股明細 |
| TPEx | `openapi/v1/tpex_mainboard_daily_close_quotes` | 上櫃每日收盤行情 | ✅ 400+ 筆（含 ETF/債券基金,需過濾出純股票） |
| TPEx | `openapi/v1/tpex_mainboard_peratio_analysis` | 上櫃 PE／殖利率／PBR | ✅ **1,000+ 檔**，股票代碼範圍 1240–6435 |
| TPEx | `openapi/v1/tpex_3insti_*`（三大法人） | 上櫃三大法人買賣超 | 端點存在（Swagger 確認），尚未逐一實測回傳內容 |

**代號格式**：TWSE/TPEx 回傳的股票代碼（如 `1101`、`2409`）與 FinMind 的 `stock_id` 格式完全一致，
不需要額外的代號對照表。

**已確認的硬限制**：
- 無歷史日期參數 → 只能「即時輪詢、自建歷史」，無法瞬間回填過去
- TPEx 的行情端點混雜 ETF／債券基金／個股，需要用代碼規則或 `TaiwanStockInfo` 產業對照表過濾出純股票
- 尚未驗證的部分：TPEx 三大法人端點的實際回傳格式、是否有隱性請求頻率限制（雖然官方文件未寫,
  建議比照 TWSE 一天一次的呼叫節奏,不要短時間內高頻打)

## 3. 架構設計：兩層式，職責徹底分離

```
┌─────────────────────────────────────────────────────────────┐
│ 第一層：廣度層 (TWSE/TPEx 官方開放資料,免費無限流,只有「今天」)   │
│   → 全市場 ~1,700-1,800 檔,股價 + PE/PBR/殖利率 + 法人流向      │
│   → 每天一次呼叫 (6 支端點),存進新的本機快照庫 (自建歷史,逐日累積)│
│   → 用途:粗篩,產生候選名單 (見 §4 Phase 2 的篩選規則設計)        │
└───────────────────────────┬─────────────────────────────────┘
                             │ 候選名單 (預估 50-200 檔,規模與現有
                             │ watchlist/測試池同量級)
                             ▼
┌─────────────────────────────────────────────────────────────┐
│ 第二層:深度層 (FinMind,現有 PIT 五維評分管線,完全不變)          │
│   → 只對候選名單 + 你的 watchlist.txt 跑完整 8 資料集抓取        │
│   → 檔數小 (50-200),600次/小時額度綽綽有餘,日常步調不變          │
│   → 沿用 build_pit_stockdata / ScoringManager / advisor        │
└─────────────────────────────────────────────────────────────┘
```

**核心設計原則**：第一層只做「便宜、廣、淺」的排除法；真正花 FinMind 額度的深度評分，
永遠只對「已經被粗篩留下來的少數股票」執行。全市場的廣度不會讓 FinMind 的每日負擔增加。

## 4. 分階段實施計畫

### Phase 0 — 可行性驗證（✅ 本規劃書已完成）
六端點實測、代號格式確認、歷史限制確認。**產出**：本文件。

### Phase 1 — 建立「廣度層」每日快照收集器
- 新增 `core/market_snapshot_provider.py`：封裝 TWSE/TPEx 六支端點，正規化成
  `stock_id / date / close / per / pbr / dividend_yield / foreign_net / trust_net / dealer_net`
  等與 FinMind 慣例一致的欄位。
- 新增獨立的 Parquet 累積庫（例如 `<FINMIND_CACHE>/MarketSnapshot/`，與現有 `finmind_cache` 分開，
  避免混淆「PIT 嚴格歷史」與「這個新的、只能逐日累積的快照」兩種資料的信任等級）。
- 接進現有 `scripts/daily_auto_update.bat`：每天收盤後跑一次（六次 API 呼叫，秒級完成），
  append 進快照庫。
- 待辦：實測 TPEx 三大法人端點格式、確認 TPEx 行情端點如何過濾出純股票（可能需要對照
  `TaiwanStockInfo` 或 TPEx 自己的股票清單）。
- **產出**：每天自動多一列「全市場快照」，累積中，尚不能拿來驗證任何排序規則（樣本太少）。

### Phase 2 — 粗篩規則設計與「回溯驗證」（決定性關卡）
這一步刻意跟 Phase 1 解耦，因為 Phase 1 的資料還沒有歷史深度可以驗證。改用你已經信任、
已有 2019 至今歷史的 FinMind PIT 管線，在一個**比 45 檔測試池更大、但仍受控**的 FinMind 抽樣
（例如 100–200 檔，一次性抓取、在 API 額度內）上，測試候選的粗篩規則本身有沒有排序力：

- 候選規則範例：PE 歷史分位低 + 近期動能轉強 + 法人近期淨買超為正（三選二或加權）
- 用現有工具驗證：仿照 `scripts/factor_experiments.py` 的方法論，對這個規則跑
  Rank IC / 市場中性多空 / `--cycle` 式的三週期穩健性
- **通過門檻**：與 v4.4 訊號篩選同一套標準——全期與 2022 空頭段都不能明顯變差
- **不通過就停在這裡**：若粗篩規則本身沒有排序力，Phase 3/4 不值得做（廣度層再大也是雜訊）

### Phase 3 — 接線：廣度層快照 → 粗篩 → 候選名單 → FinMind 深度評分
- Phase 2 驗證通過後，把規則套用在 Phase 1 累積的即時全市場快照上，產生每日候選名單
  （預估 50–200 檔）
- `build_cache.py` 新增類似 `--screen-universe` 的模式：symbols = `watchlist.txt` ∪ 當日候選名單
- 候選名單股票才觸發 FinMind 抓取（沿用現有 8 資料集 PIT pipeline），其餘全市場股票
  只停留在第一層快照，不耗 FinMind 額度
- **產出**：網頁選股從「你手動維護的名單」升級成「你的名單 + 系統每天從全市場撈出的候選」

### Phase 4 — 長期複驗（需要時間，不能跳過）
- Phase 1 的快照庫累積滿 6–12 個月後，才有足夠樣本回頭檢驗「這套粗篩規則在真實的
  TWSE/TPEx 即時資料上，挑出來的候選名單後續表現是否真的好」（用候選名單股票的 FinMind PIT
  評分結果對照，補上 Phase 2 用 FinMind 抽樣驗證時無法涵蓋的「即時資料本身的雜訊/延遲」因素）
- 這階段本質是「先上線、邊跑邊等資料長出來」，不是可以壓縮工期的部分

## 5. 風險與待解問題

| 項目 | 風險 | 因應 |
|---|---|---|
| TPEx 法人端點未實測 | 格式/覆蓋率未知 | Phase 1 開工前先補測 |
| TPEx 行情含 ETF/基金雜訊 | 粗篩誤把非股票商品排進候選 | 用代碼規則或既有產業對照表過濾 |
| 無歷史回填 | 粗篩規則的「即時資料版」要等 6-12 月才能真驗證 | Phase 2 先用 FinMind 抽樣驗證規則邏輯本身,降低這個風險 |
| 官方端點無文件化限流 | 高頻呼叫可能被視為異常流量 | 比照現有排程,一天一次,不做高頻輪詢 |
| 資料清洗水準低於 FinMind | 無 `_back_adjust` 這類除權息/分割接平處理 | 廣度層只做粗篩排序,不進 PIT 回測計分,可容忍;真正計分仍交給 FinMind 那一層 |
| 存活者偏誤 | 下市股票會直接從快照消失 | 廣度層是「即時前瞻使用」,此問題主要影響回溯驗證,天生比歷史回填資料集溫和 |

## 6. 決策點

這個方案值得做的前提，取決於你要不要真正跳脫「手動維護 watchlist」、換成「系統每天自動從
全市場撈候選」。若答案是要，建議順序做 Phase 1→2→3，Phase 2 是關鍵停損點——先花小成本
（在 FinMind 額度內抽樣驗證規則），再決定要不要投入 Phase 1 的資料收集器開發。

若你目前仍偏好「維持 watchlist 手動選股 + 系統深度分析」的既有流程（前面幾輪對話中你提過
這才是實際用法），這份規劃書可以先當作**文件化的可行方案**保留，不用立刻動工。
